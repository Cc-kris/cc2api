## 2. 问题闭环

线上 `0.2.43` 已修复 Responses 图片桥接直连、重试去重、输入图保留和上一轮图片编辑等问题，但仍存在一个关键缺口：桥接幂等把“自动重试”和“用户再次发起生成”混在同一判断里，并且成功状态写入使用请求上下文。客户端断开后，成功状态可能写库失败，后续同内容请求会被当作重复请求压掉，返回空的 completed 响应，前端继续展示上一张图片，造成“第二次给了同样一张图”的表现。

本次改造目标是把图片桥接做成完整闭环：真实新请求必须重新上游生成；自动重试必须复用已成功结果或明确告知仍在处理；任何重复压制都不能返回空成功。

## 3. 实现目标

| 目标 | 落地结果 |
|---|---|
| 新请求不误杀 | 同提示词、同会话、不同客户端请求 ID 的两次生成视为两次真实请求 |
| 自动重试可回放 | 同客户端请求 ID 的重复请求命中已成功记录时返回完整图片结果 |
| 处理中不伪成功 | 首次请求未完成时重复请求返回明确处理中状态和 `Retry-After`，不返回空 completed |
| 客户端断开有闭环 | 上游成功后使用后台短超时上下文保存结果，不受客户端断开影响 |
| 图片编辑稳定 | 输入图、上一轮生图结果、mask、多图输入继续进入 edits 路径 |
| 错误可恢复 | 上游失败、读取失败、空结果进入短暂可重试状态 |
| 可观测可验证 | 日志、测试和返回内容能区分 replay、processing、new generation |

## 4. 场景边界

| 场景 | 系统行为 |
|---|---|
| Responses 请求显式选择图片工具 | 桥接到 Images generations/edits |
| Responses 顶层模型为图片模型 | 转换为图片工具请求后桥接 |
| 纯文本 Responses | 不进入图片桥接 |
| 图片生成无输入图 | 请求 `/v1/images/generations` |
| 图片编辑有输入图、上一轮结果或 mask | 请求 `/v1/images/edits` |
| 同请求 ID 自动重试 | 不重复上游；成功则回放，处理中则返回处理中 |
| 不同请求 ID 的再次生成 | 重新请求上游，生成新的图片 |
| 上游成功但客户端断开 | 保存完整结果和使用量；后续同请求 ID 可回放 |
| 上游失败或空结果 | 标记短暂可重试，不写成功结果 |
| 流式请求 | 回放完整 SSE 事件，不输出空 completed |
| 非流式请求 | 回放完整 JSON 响应，不输出空 output |

## 5. 技术改造清单

### 5.1 幂等身份改造

当前逻辑优先使用 `turn_id`，导致同一会话内两次真实请求被合并。改造后身份规则如下：

| 输入条件 | 幂等身份 |
|---|---|
| 有 `X-Client-Request-ID` | `client:<client_request_id>` |
| 无客户端请求 ID 且有 `turn_id` | `turn:<turn_id>` |
| 二者都没有 | 不启用桥接幂等，仅按普通请求处理 |

`turn_id` 保留在指纹与日志中用于排查，不再覆盖客户端请求 ID。

### 5.2 回放载荷改造

复用现有 `idempotency_records.response_body` 字段保存图片桥接回放包，不新增数据库表。回放包包含：

| 字段 | 说明 |
|---|---|
| `kind` | 固定为 `responses_image_bridge_replay` |
| `version` | 回放格式版本 |
| `status_code` | HTTP 状态码 |
| `content_type` | `application/json` 或 `text/event-stream` |
| `stream` | 是否为流式响应 |
| `body` | 完整客户端响应体 |
| `image_count` | 图片数量 |
| `created_at` | 生成时间 |

该方案在不迁移数据库的情况下满足“成功后可完整回放”的闭环要求。

### 5.3 响应构建改造

将原先“边构建边写响应”的逻辑拆成两步：

1. 先构建完整 JSON/SSE 响应体；
2. 成功保存幂等回放包；
3. 再写给客户端。

这样可以保证客户端断开不会导致结果没有保存。

### 5.4 后台状态写入改造

`markSucceeded` 和 `markFailedRetryable` 不再使用请求上下文，统一使用 `context.Background()` 加短超时执行。客户端断开、浏览器取消、代理 broken pipe 不影响最终状态落库。

### 5.5 重复请求响应改造

| 记录状态 | 返回行为 |
|---|---|
| `succeeded` 且有完整回放包 | 原样回放 JSON/SSE 图片结果 |
| `succeeded` 但缺少回放包 | 返回明确失败状态，说明结果不可回放，不伪装成功 |
| `processing` | 返回处理中错误语义和 `Retry-After` |
| `failed_retryable` | 返回重试退避语义和 `Retry-After` |
| 指纹冲突 | 返回冲突语义，不重复上游 |

### 5.6 图片输入解析保留

继续保留以下能力：

- `input_image.image_url`
- `image_url.url`
- `url`
- `data`
- `b64_json`
- `image_generation_call.result`
- `input_image_mask`
- 多图输入去重

本次改造不改变图片输入解析主路径，只补齐重试与回放生命周期。

### 5.7 错误与降级策略

| 错误类型 | 处理方式 |
|---|---|
| 上游请求失败 | 标记 `failed_retryable`，短暂退避 |
| 上游读取失败 | 标记 `failed_retryable` |
| 上游 4xx/5xx | 记录状态码和错误摘要，进入既有 failover |
| 上游成功但无图片 | 标记 `failed_retryable`，返回错误 |
| 客户端写入失败 | 不回滚成功状态；同请求 ID 后续可回放 |
| 幂等存储不可用 | 记录日志，桥接请求继续按普通路径执行 |

## 6. 测试方案

| 测试类型 | 覆盖内容 |
|---|---|
| 正向生成 | 非流式/流式图片生成返回完整 output |
| 正向编辑 | 输入图、上一轮生图结果进入 edits |
| 多图结果 | `n>1` 返回多张图片并计数正确 |
| 新请求重复提示词 | 不同客户端请求 ID 重新请求上游 |
| 自动重试回放 | 相同客户端请求 ID 不重复上游，返回完整图片 |
| 处理中重复 | 未完成时重复请求返回处理中和 `Retry-After` |
| 指纹冲突 | 同请求 ID 不同 payload 返回冲突语义 |
| 上游错误 | 上游失败进入可重试状态 |
| 上游空结果 | 不写成功，返回错误 |
| 客户端断开 | 结果先保存，写失败后仍可回放 |
| 多步骤编辑 | 生成结果作为下一步编辑输入 |
