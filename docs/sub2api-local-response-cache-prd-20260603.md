# Sub2API 本地精确 Response Cache PRD

版本：2026-06-03-01
适用仓库：Cc-kris/sub2apis
发布目标：第一版保守上线，降低重复请求的上游调用成本。

## 1. 背景

当前 sub2api 已记录上游返回的 cache creation / cache read token，并具备上游缓存相关计费字段，但缺少站长侧本地 Response Cache：当用户短时间内重复提交完全相同请求时，系统仍会请求上游账号，产生上游费用与账号压力。

线上机器当前配置为 2 核、3.8GB 内存、Redis 当前占用约 2.9MB、数据库约 942MB、最近 24 小时约 2406 次请求。该规模适合先上线保守缓存策略：API Key 内精确缓存、短 TTL、限制单条响应大小、Redis 异常自动降级。

## 2. 目标

1. 对 OpenAI 平台文本请求提供本地精确 Response Cache，覆盖非流式 JSON 与流式 SSE。
2. 缓存命中时直接返回上次响应，不请求上游账号。
3. 缓存范围默认限定在同一个 API Key、同一个分组、同一个模型、同一请求参数内。
4. API Key 切换分组后不命中旧分组缓存。
5. Redis 不可用、缓存读写失败、请求不适合缓存时，自动走原有上游转发链路。
6. 第一版不改变用户请求语义、不做语义相似缓存、不跨用户共享缓存。

## 3. 非目标

1. 不做跨用户、跨分组、全站共享缓存。
2. 不做跨请求语义相似流式合并；流式缓存采用原始 SSE 捕获与回放。
3. 不缓存图片、语音、文件、实时 WebSocket 请求。
4. 不缓存工具调用、函数调用、带副作用请求。
5. 不实现 W-TinyLFU 淘汰策略第一版。
6. 不把完整 prompt 或完整 response 存入 PostgreSQL。

## 4. 用户角色

| 角色 | 需求 |
|---|---|
| 站长 | 降低重复请求的上游成本，控制缓存容量和风险 |
| 普通用户 | 重复请求响应更快，费用更低或不增加上游消耗 |
| 运维人员 | 能通过日志/header 判断请求是否命中缓存 |

## 5. 功能范围

### 5.1 支持接口

第一版支持 OpenAI 平台下列请求，包含非流式 JSON 与流式 SSE：

- `/v1/responses`
- `/v1/chat/completions`
- `/v1/messages` 通过 OpenAI 平台转发的兼容请求

### 5.2 命中条件

缓存 key 必须包含：

- API Key ID
- Group ID
- 接口类型
- 平台
- 请求模型
- 标准化后的请求 body hash

同一 API Key 切换分组后，因为 group_id 不同，不能命中旧缓存。

### 5.3 默认跳过条件

以下请求不缓存：

- 图片生成意图
- WebSocket 请求
- 请求体超过最大可缓存请求体大小
- 响应体超过最大可缓存响应体大小
- 请求体包含 tools / tool_choice / function_call / functions
- 请求体包含敏感字段关键词
- temperature 大于配置阈值
- 客户端 header 指定绕过缓存

### 5.4 返回标记

命中缓存时响应 header 返回：

- `X-Sub2API-Cache: hit`

未命中并正常走上游时，成功写入缓存后可以返回：

- `X-Sub2API-Cache: miss`

不适合缓存时：

- `X-Sub2API-Cache: bypass`

### 5.5 默认运行参数

| 参数 | 默认值 |
|---|---:|
| 总开关 | 关闭，可在系统设置 → 通用设置打开/关闭 |
| 默认 TTL | 10 分钟 |
| 单条响应最大缓存 | 512KB |
| 单条请求体最大缓存 | 256KB |
| temperature 最大值 | 0.3 |
| 缓存范围 | API Key + Group |

## 6. 验收标准

1. 功能默认关闭，不影响现有请求链路。
2. 开启后，同一 API Key、同一分组、同一请求第二次命中缓存。
3. API Key 切换分组后不会命中旧分组缓存。
4. stream=true 请求首次正常转发并捕获完整 SSE，成功结束后写入缓存；再次请求命中后回放 SSE。
5. tools/function_call 请求不缓存。
6. Redis 读写失败时请求仍能正常转发上游。
7. 自动化测试覆盖 key 生成、跳过规则、缓存读写、命中响应。

## 7. 系统设置要求

后台「系统设置 → 通用设置」必须新增本地响应缓存开关。开关默认关闭；保存后立即影响后续请求。第一版只暴露总开关，容量、TTL、大小阈值使用后端安全默认值，后续再扩展高级配置。

## 8. 流式缓存要求

流式请求首次不改变原始转发行为：服务端边接收上游 SSE，边转发给客户端，同时捕获已成功写出的 SSE 原文。只有当上游正常结束、HTTP 状态为 200、响应大小不超过限制时才写入缓存。命中缓存时，按原始 Content-Type 回放缓存的 SSE body，并返回 `X-Sub2API-Cache: hit`。
